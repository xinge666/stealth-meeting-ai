# 大语言模型与RAG技术详解

## 第一部分：语言模型基础

语言模型是自然语言处理的核心。它的基本任务是根据部分文本预测下一个词。最早的语言模型基于统计，例如 N-gram 模型。然而，这一类模型受限于数据稀疏问题，不能很好地处理长距离的依赖关系。

随着深度学习的发展，神经网络语言模型（NNLM）逐渐兴起。RNN、LSTM等模型在序列建模上取得了巨大成功。而在2017年，Google 提出了 Transformer 架构。Transformer 引入了自注意力机制（Self-Attention），使得模型能够并行计算，并有效捕捉长距离依赖。

在 Transformer 中，计算注意力时需要除以 dk 的平方根。为什么要有这一步缩放操作呢？这是由于点积的数值大小会随着维度 dk 的增大而变大，进而导致 softmax 函数进入梯度极小的区域。为了保证梯度稳定，除以根号 dk 可以使点积结果的方差维持在1附近。

## 第二部分：生成与检索增强（RAG）

生成式模型由于依赖训练语料，往往存在知识幻觉（Hallucination）问题，也无法实时获取最新的外部知识。为了解决这一问题，检索增强生成（Retrieval-Augmented Generation, RAG）应运而生。

RAG 的基本流程如下：
1. 文档切片（Chunking）：将外部知识库中的长文档切割成固定大小的文本块。这个文本块的大小可以通过 Token 数量来控制。比如，使用 64 token 作为一个分片。
2. 向量化（Embedding）：通过专门的语义向量模型（例如 sentence-transformer 的 moka-ai/m3e-base），将这些分片转换成高维向量，并存储到向量数据库中。
3. 检索（Retrieval）：当用户提出问题时，同样将查询内容向量化，使用余弦相似度等度量方法，从向量库中匹配最相关的若干个文本块。
4. 增强生成（Generation）：把检索到的相关文本块连同用户的问题一起送给大语言模型，让模型基于这些参考资料给出最终答案。为了提供更多的上下文，检索时甚至可以连带输出命中分片的上下若干个相邻分片，以保证语意连贯性。

## 第三部分：系统设计与工程落地

当你准备系统设计面试时，需要考虑以下几个方面：
- 用户规模和吞吐量（QPS估算）
- 数据存储层的选择（关系型数据库，NoSQL，还是向量库）
- 缓存策略（Redis，Memcached）来应对热点数据
- 服务解耦（消息队列如 Kafka）以应对突发流量
- 架构的横向和纵向扩展能力

在实际落地 RAG 系统时，要面临向量召回效率、排队耗时、LLM 首字响应时间等工程挑战。
