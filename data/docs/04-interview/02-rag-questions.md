## 第一部分：RAG 核心原理（必考）

**Q1: 请解释 RAG 的工作原理。与直接对 LLM 进行微调相比，RAG 主要解决了什么问题？有哪些优势？**
- 难度：⭐⭐
- 岗位：通用
- 标签：#RAG #检索增强 #核心概念
- 公司：字节、阿里、腾讯（高频）

**【参考答案】**
**工作原理**：RAG（Retrieval-Augmented Generation 检索增强生成）结合了信息检索和生成式 AI。系统首先基于用户提问（Query）在外部知识库中检索高度相关的文档切块，然后将召回的上下文事实与用户的原始提问结合拼接成新的 prompt，最后交给大语言模型（LLM）基于给定的上下文生成准确的回答。
**解决的问题**：
1. **幻觉（Hallucination）**：通过提供真实可靠的参考资料，大幅度限制大模型无中生有的能力。
2. **知识滞后问题**：大模型的知识停留在训练数据截止的日期，RAG 能够实时引入最新的外部数据，无需重训。
3. **企业私有知识的安全**：企业可以用开源 LLM 搭配内部知识库进行 RAG，从而实现私有数据不出域，解决了通用模型不懂特定领域黑话或私有规范的问题。
**对比微调的优势**：极低的成本（减少了昂贵的算力与微调数据清洗）、优异的可解释性（回答的每一部分溯源可查）、更新灵活度满分（只需增删改数据库内容即实时生效，而微调极难精准地抹除某部分知识）。

**Q2: 一个完整的 RAG 流水线包含哪些关键步骤？请从数据准备到最终生成，详细描述整个过程。**
- 难度：⭐⭐
- 岗位：通用
- 标签：#RAG #流程 #系统设计

**【参考答案】**
完整的系统包含三大步骤：
1. **数据准备与注入（Data Indexing）**：
   - **抽取与解析**：将各种类型的文档（PDF, HTML, Word）处理并从中提取干净的纯文本、图片、表格要素。
   - **分块（Chunking）**：将超长文档切分成大小合理、有结构逻辑的文本片段。
   - **向量化（Embedding）**：用模型将所有的文本分块转变为多维密集的向量。
   - **入库存储**：把生成的向量及其原文片段、对应的元数据（时间、作者等）存储到向量数据库中。
2. **检索（Retrieval）**：
   - 接收用户 Query 并用同一套 Embedding 模型将其转化为向量。
   - 在向量数据库中执行相似性搜索（加上可能的 BM25 稀疏检索混排），召回最相似的 Top-K 个知识块。
   - (可选) 重排序（Reranking）：通过大模型或专门的 Cross-Encoder 对初步召回的 Top-K 进行精细二次打分与排序。
3. **生成（Generation）**：
   - 将用户 Query 以及重排序后的前几名知识片段组合并塞入 System Prompt。
   - 请求推理端执行大模型，大模型阅读参考内容，最后按照要求梳理格式吐出直接回答。

**Q3: 在构建知识库时，文本切块策略至关重要。你会如何选择合适的切块大小和重叠长度？这背后有什么权衡？**
- 难度：⭐⭐
- 岗位：通用
- 标签：#RAG #文档处理 #切块策略

**【参考答案】**
**如何选择**：
- **切块大小（Chunk Size）**：取决于业务提问多属于深挖细节的孤立知识，还是要求总结归纳的长文本。一般来说 512 到 1024 左右较常见；如果嵌入模型和 LLM 的上下文可以承载，特定长图文可以放宽。
- **重叠长度（Overlap）**：为了防止段落意义在切片边缘被硬挤断无法找回上下文，一般会设置块大小的 10%-20% 作为重叠缓存区。
**背后的权衡**：
- **较小的 chunk**：向量嵌入出来的语义特征更明确，信噪比高且容易高分召回。缺点是切断了上下文，大篇幅生成易失去连接，同时扩大了向量数据库的条目规模。
- **较大的 chunk**：附带完整的原汁原味上下文信息，方便 LLM 进行总结推理。缺点是一块文本包含多个话题导致检索时不突出，噪音也会较多，并消耗 LLM Token；
- **重叠长度的权衡**：重叠越多上下文越连贯，越不易遗漏关键过渡句；但系统数据冗余指数级增加，对检索器存在一定的命中混淆。

**Q4: 如何选择一个合适的嵌入模型？评估一个 Embedding 模型的好坏有哪些指标？**
- 难度：⭐⭐
- 岗位：通用
- 标签：#RAG #Embedding #模型选择

**【参考答案】**
**选择合适 Embedding 模型的考量维度**：
1. **语言能力**：看其是在纯英文语料还是多语言语料上训练的，必须和目标应用语言适配（如 BGE-M3，Jina，OpenAI）。
2. **输入上下文长度支持**：部分老旧模型只支持 512 Tokens 的输入，如果遇到长篇大块输入会发生截断丢失，需要选用支持 8k 甚至更长的模型。
3. **成本和响应延迟要求**：高维度模型（>2048维）精准度可能略好但是极其昂贵且耗时，低维度（如 384/768）更轻量快巧。
**评估好坏的量化指标**：
大部分使用评测基准（如 MTEB）。基于标注数据集的核心检索指标为：
- **命中率（Hit Rate @ K）**：召回的前 K 篇文档中能命中真实验证集的比例。
- **平均倒数排名（MRR, Mean Reciprocal Rank）**：关注第一个正确的、相关文档出现在最终召回排序表里面的位置（越靠前，倒数的分数越趋近1）。
- **NDCG @ K（归一化折损累计增益）**：考虑召回的前 K 个排名的位次分级权重。

## 第二部分：检索优化（算法岗重点）

**Q5: 除了基础的向量检索，你还知道哪些可以提升 RAG 检索质量的技术？**
- 难度：⭐⭐⭐
- 岗位：算法岗、开发岗通用
- 标签：#RAG #检索优化 #混合检索
- 公司：字节、阿里（高频）

**【参考答案】**
除了 Dense Vector（稠密向量检索）外，常见增强手段有：
1. **混合检索（Hybrid Search）**：把基于关键字统计的稀疏检索（Sparse，如 BM25）和基于深层语义的稠密向量检索结合，使用 RRF（倒数排名融合）将两路候选列表进行综合打分。（精准解决实体名称或特殊黑话错漏问题）。
2. **假设性文档嵌入（HyDE, Hypothetical Document Embeddings）**：先让 LLM 把用户的短破 query 顺口"编撰/猜测"成一篇答案，然后再对这篇虚拟长文 Embedding 后去做真实长文档相似度匹配，能有效对齐空间特征。
3. **多粒度路由与子查询分解（Query Routing & Decomposition）**：把复杂复合的问题使用 LLM 降维打击切分成短单句进行分别多线程检索，最后将子检索结果大聚合。
4. **粗召回 + 二次深度重排精排（Rerank）**：前面的步骤先粗筛前百项，再通过对上下文强关联的交叉编码器进行极耗时的二阶段精细打分卡口。
5. **父子文档合并（Parent-Child / Auto-merge DocumentRetrieval）**：入库时切成短 chunk，命中极小 chunk 后，连带将当时附带的包含它本身的更大长篇父节点章节捞出来喂给生成器。

**Q6: 请解释"Lost in the Middle"问题。它描述了 RAG 中的什么现象？有什么方法可以缓解这个问题？**
- 难度：⭐⭐⭐
- 岗位：算法岗重点
- 标签：#RAG #长文本 #优化

**【参考答案】**
**现象含义**：
“Lost in the Middle”（中间迷失）指的是当我们向大语言模型的上下文中一次性塞入过多的检索文档片段时，模型会呈现与人类十分相似的首因效应与近因效应，即：能够完美参考放置在最开头或最末尾的有效信息提取并回答，但对于掩埋在中间大段文本中的核心信息视而不见。
**环节/应对方案**：
1. **文档序列重排（Reordering）**：将所有通过精排分数高的资料人为进行位次重新排列。比如选最高分放第一位，次高分放末尾位，分数居中的全都放置在中间区域，迎合它的机制特征。
2. **缩减内容提供量（压缩与卡口）**：严控入送 Top-K 的数量。同时还可以加上基于抽取式 Summarization 模型或者小 LLM 对每一段信息预先进行不相干废话剪枝和压缩。
3. **长上下文强化模型替代**：不再使用普通窗口模型，而换上针对超长上下文做了“大海捞针”针对性优化微调的模型（比如采用动态 RoPE 或者特定强化版长文结构）。

**Q7: 在什么场景下，你会选择使用图数据库或知识图谱来增强或替代传统的向量数据库检索？**
- 难度：⭐⭐⭐
- 岗位：算法岗重点
- 标签：#GraphRAG #知识图谱 #检索策略

**【参考答案】**
**GraphRAG / 知识图谱 特别适用的痛点场景**：
1. **跨文档的强关联多跳推理（Multi-hop Reasoning）**：比如：“这个投资项目的母公司的最高层是在哪家学校任教的？” 纯向量常常因碎片化提取缺失这几层链条。而图谱有明确的点边互联关系，可以通过实体一节节遍历过去。
2. **全局系统性理解与摘要**：“请总结全库文件中所有的财务合规风险体系。” 向量很难跨千篇文档找宏大结构关联，但知识引擎通过层次化图发现和社区划分（Graph Partition）能层级往上提供宏大叙事。
3. **低错误低幻觉（Zero-Tolerance Domain）**：医学判别、严格法规、司法裁判条文关联，要求 100% 精确的业务因果链，而依靠余弦相似这种非受控统计特征是不被信任的。知识图谱结构确定，能实现精准的事实穿透。

## 第三部分：评估与优化（通用）

**Q8: 如何全面地评估一个 RAG 系统的性能？请分别从检索和生成两个阶段提出评估指标。**
- 难度：⭐⭐⭐
- 岗位：通用
- 标签：#RAG #评估 #指标设计

**【参考答案】**
常见的评估范式（如 Ragas / TruLens 等引入大模型充当裁判LLM-as-a-Judge 的三元组分析法）将其拆解：
**第一阶段：对”搜索检索阶段“评估（Retrieval Evaluation）**
- **上下文相关度（Context Relevance）**：针对用户的 query，检索回的大把内容必须和 query 真正有关，废话杂音少（侧重精准）。
- 召回率/准确率（Recall / Precision @ K）、NDCG 等经典搜索工程评估集。
**第二阶段：对”生成阶段“评估（Generation Evaluation）**
- **忠实度/零幻觉度（Faithfulness / Groundedness）**：强制保证 LLM 最后写出的生成回答所主张的每个事实实体陈诉，这必须且仅来自于提供的 context，如果有自编乱造的场外推断就是负分。
- **答案直接相关度（Answer Relevance）**：针对问题没有绕弯子套话，直指人心没有缺漏。如果参考文档无，它是否直接声明无法回答？
- **准确全面性（Correctness）**：如果有标注好的 ground truth 真值，与真值的信息量重合度有多少。

**Q9: 传统的 RAG 流程是"先检索后生成"，你是否了解一些更复杂的 RAG 范式，比如在生成过程中进行多次检索或自适应检索？**
- 难度：⭐⭐⭐⭐
- 岗位：算法岗重点
- 标签：#RAG #高级技术 #自适应检索

**【参考答案】**
这是从普通 RAG 向高级 Agent 过渡的方法：
1. **FLARE（前瞻性主动检索）**：一边流式生成句子一边动态侦测概率衰退；当下一个将生成的预测词/实体模型置信度（Confidence）极低时，它就会暂停生成把它当作新的 query 抛去检索平台找答案再回来继续生成。
2. **Self-RAG（具有反思纠错能力的自身检验系统）**：通过微调的基座大模型（或设计巧妙的提示词）让回答者在遇到某个模块信息自己带有诸如 `[是否需要检索]、[相关吗]、[支持吗]、[完美回答吗]` 的 Critic 思考标签位；实现全自动的检索、自我校验评价与主动修改。
3. **Iterative RAG (迭代检索)**：先查出第一个知识线索，将其加入到当前上下文后重新基于当前线索扩展构造出一个新 query。反复此回环直到积累的知识拼图宣告完成解答。
4. **Agentic RAG（类智能体RAG架构）**：以大模型 ReAct 为中枢控制器，它视检索数据库与其他所有业务执行 API 一样，作为一项调用（Call）工具，它自己编写调度逻辑拆解问题后依次收集执行。

## 第四部分：工程实践（开发岗重点）

**Q10: RAG 系统在实际部署中可能面临哪些挑战？**
- 难度：⭐⭐⭐
- 岗位：开发岗重点
- 标签：#RAG #部署 #工程实践
- 公司：字节、阿里（高频）

**【参考答案】**
工程开发挑战在于落地时不再是算法玩具：
1. **多租户权限硬隔离**：2B 场景中向量数据库里装着几万员工甚至多公司的绝密文档，检索请求下沉到数据库的时候必须做严格 Metadata 过滤，不可跨界召回别人权限外内容并防止 Prompt 越权注入攻击。
2. **耗时深坑吞噬（Latency SLA）**：RAG 包含模型请求重写、多并发 IO 查索引库、GPU服务器做重排模型推理、流式回复网络传输等诸多节点。系统延迟常大幅超过业务方规定的几十毫秒，迫切需要 Semantic Cache、并行异步请求等手段打补丁提速。
3. **复杂格式及非结构格式文档解析（Parsing Nightmare）**：算法做得再花哨都卡在这。业务中的多栏混排论文 PDF、海量印章横纵交错表格，甚至图像格式图纸，让基于光栅甚至传统 OCR 极容易发生块合并错乱与表头对齐崩溃，需做定制抽取大模型攻坚。
4. **知识热更新同步的吞吐量**：文件库删除、重命名、新增部分段落后的脏数据污染，或者需要分钟级甚至秒级的爬虫文档索引热注入能力保证新鲜度。

**Q11: 了解搜索系统吗？和 RAG 有什么区别？**
- 难度：⭐⭐
- 岗位：开发岗重点
- 标签：#RAG #搜索系统 #对比分析

**【参考答案】**
**核心定位存在区别**：
- **传统的检索引擎/搜索引擎系统（ElasticSearch等类）**：主要价值立足于**重在定位排序和引流**，给用户呈现的是一堆带有标题高亮加粗粗暴摘要链接；由系统另一端活人肉眼浏览判断并做真正的提炼萃取；其基底多是以倒排索引关键词切词为脉络；
- **RAG 系统架构**：主要价值落脚在一口喂到嘴边的**直接生成回答**；它的技术构成其实是在搜索系统链路的顶端插了一层“带有超强深度语言阅读理解并能够用人话沟通发散的”生成器脑部模块。
**从属与升级关系**：工业最优秀的 RAG 的核心底盘恰好一定是一个非常强悍的多模混合搜索系统；RAG 将搜索结果变现的门槛大大降低。

**Q12: 知道或者使用过哪些开源 RAG 框架比如 Ragflow？如何选择合适场景？**
- 难度：⭐⭐
- 岗位：开发岗重点
- 标签：#RAG #框架选型 #工程实践

**【参考答案】**
1. **Langchain / LlamaIndex**：元老基建，模块化散件积木非常多而且社区生态插件多。但是由于抽象粒度过于繁多和嵌套深度难以掌控，当前工业落地环境很多企业开始抛弃它转手开发自研逻辑，或只用于起步快速验证 Demo。
2. **Ragflow**：最极致专精于 **Deep Document Understanding（深度文档理解结构解析）**的分块痛点；内置包含极其完备优异的智能视觉 OCR 与版面分析深度算法（Layout Analysis）；
   - **合适场景**：金融业审计底稿、法律行研长跨图论文居多等，极具排版格式硬骨头的刚需场景。
3. **Dify / FastGPT / Qwen-Agent**：是一套带着漂亮 WebUI 的完整知识编排和工作流管线（Workflow Builder）无代 / 低代码成品系统；
   - **合适场景**：帮助中小企业内部运维或者产品经理利用知识库火速搭起业务 Agent（能外挂 API 和发微信机器人）。拿来即用交付落地的极好阵地。

## 第五部分：RAG 进阶技术（真题补充）

**Q13: 构建向量检索库时如何处理时间衰减对召回的影响？**
- 难度：⭐⭐⭐
- 岗位：算法岗重点
- 标签：#时间衰减 #召回优化
- 公司：字节(真题)

**【参考答案】**
业务痛点：很多新闻问答或者价格策略问答中，新鲜的旧文档相似度可以相同但价值必然递减。可以有两种维系手段：
1. **分数公式融合调权法**：在基础打分排序上叠加快慢机制。对相似度 $S_向量$ 给它加上或乘上结合此块入库日期或者文件时间戳设计的指数衰减项函数（如 $S_{Total} = \alpha * S_v + (1-\alpha) * \exp(-\lambda * \Delta t)$）。使得时间更久打分大打折扣。
2. **硬性倒排过滤截断**：让 LLM 做 Query 解析，遇到针对带有“最新”，“刚刚”，“目前发布”意图时直接解析出元字段表达式去 DB 过滤，把时间窗封死在 `datetime >= 前几天短周期` 范围内后再进行纯向量召回。在使用交叉打分等大模型深度 Rerank 环节也可以把时间信息连带注入强喂给它训练。

**Q14: RAG中知识库搭建，对知识库的文件文档进行动态增量更新，怎么来避免新旧文档的分布不一致导致的检索偏差问题？**
- 难度：⭐⭐⭐
- 岗位：算法岗重点
- 标签：#增量更新 #分布一致性
- 公司：美团(真题)

**【参考答案】**
如果平台经过演进替换了基础的 Embedding 大模型或者因文档风格跨界极大，会导致向量表征分布不均互相串味：
1. **表征一致化（Representation Alignment/Distillation）**：面对模型升级更新不敢一刀切时，通过对新老模型跑出的锚点抽样数据集计算出过渡的变换/投影矩阵（Procrustes Analysis），然后在查询接入时动态对老库存做平移补偿；但这往往在如今充沛算力年代是不划算的。更朴素直接的业界做法实则是对受影响存量打上待修补表记实施纯后台闲时“异步 Re-embedding 洗数据池子”，用新模型将老文档跑批覆盖。
2. **多重查询分而治之再同维重排**：维持几套并行的子数据库（如果新旧模型不可调和）。查询向量通过这两个不同的模型各自去各自的老底盘查。两路拿到 Top 候选集再喂给同一个不在乎分布偏差只要上下文即可的强大的 Cross-Encoder 判断做最后的得分统一排序截断，这属于一种在末端打平的架构弥合思路。

**Q15: RAG如果有噪声怎么办？**
- 难度：⭐⭐
- 岗位：通用
- 标签：#噪声处理 #数据质量
- 公司：字节、阿里

**【参考答案】**
RAG中的噪音（错假数据或弱相关杂料）容易影响 LLM 判断甚至加剧幻觉：
1. **源头阻断（精细 Reranker）**：卡死重排环节分数阈值红线。相关度打分不达到某个绝对信心的数值，宁可放弃传递过去返回一句“我不知道”，这远胜过给大模型放宽限制吃下毒药语料。
2. **强制增加抵抗干扰性的 Prompt (Prompt Engineering)**：给模型打免疫针。告诉模型“请仅参考给定段落回答问题，并且有些段落是完全无效干扰人的，如非必要可以剔除它”并且要求以“据资料显示XXX”结构溯源。这种对模型下达极强防御性暗示能避免它随噪声跑偏。
3. **增加文档抽取萃取器**：对 Top K 大堆文本在放入推理窗口时，垫设一层专属萃取器做先期概括提纯，只抽取实体关系，摒除发散文字噪声。
4. **提升底层模型能力评测（Robustness Eval）**：不断选取对于噪音数据抵抗力抗压鲁棒更优的底盘基座模型或单独在负例掺杂的微调微数据集加固它。

**Q16: 讲一下BM25算法原理**
- 难度：⭐⭐
- 岗位：通用
- 标签：#BM25 #稀疏检索
- 公司：字节、阿里(高频)

**【参考答案】**
BM25 (Best Matching 25) 是工业级最抗打的稀疏词法相关度信息检索底层代表算法。
它是基于早年经典的统计学模型词频-逆文档频率（**TF-IDF**）的一个数学平滑变种升级。
1. **引入词频防剧饱和曲线**：原本 TF 越高给分无限滚雪球，然而在一篇文章内一个核心词出现 10 次和出现 100 次它都是在极具聚焦该词的领域话题里（后者甚至可能是刷量）。BM25 设置了阻尼超参数 $K_1$ 平滑掉它，它呈现一个上凸极限曲线，限制了关键词轰炸篇幅权重。
2. **惩罚文档总字数（长度归一化）**：一本万字厚重百科里当然几乎必然命中个别罕见搜索术语字词，但这无法体现百科和搜索目的相关；而一篇一百字短讯里正好就有这个术语那这绝对是针对性回答。所以加入超参数 $b$ 加入文档平均字数，如果总长度高过基础平均水准会拉抵削弱打分优势。

**Q17: 是否做过意图识别？如果要做意图识别，可以怎么实现？**
- 难度：⭐⭐
- 岗位：开发岗重点
- 标签：#意图识别 #NLU
- 公司：字节、阿里

**【参考答案】**
意图识别是高级系统 Router 转路由的防线门神，常见的实现方法在性价比上逐级攀升：
1. **硬逻辑规则/正则打底**：通过极低门槛抓取特殊的句式命令甚至特定人防动作，这毫无智能可言却最坚不可摧。
2. **LLM 直接分类（Prompt Classifier）**：纯大语言模型提示词分类机制门槛最低，把分类全貌写进 prompt 让模型抛出 A B C。响应很准且支持泛化，但每一次都会吃掉大量推理费用且拉长整个请求周期的首字节等待卡口。
3. **向量距离速配（Semantic Router 机制）**：将一百多种不同意图场景事先手写或用大模型写成一千条 Ground truth 问题并转换为密集向量。后续遇到新的提问时，只要去针对这个极其小体量的特定动作库做一个几毫秒的小余弦距离查找算力就可立刻分配动作，这种做法非常快速好使。
4. **使用 NLP 独立小分类模型（Bert系分类微调）**：对于日均超千万级的调用大厂或者细分场景非常复杂的智能客服动作分拨，这是最有工程稳定保证和最低边缘耗能比的主流大做法。

**Q19: 在RAG里的"召回-过滤-生成"三段式 pipeline能细讲一下吗？**
- 难度：⭐⭐⭐
- 岗位：通用
- 标签：#Pipeline设计 #三段式
- 公司：字节(真题)

**【参考答案】**
这是一个黄金架构标准流水线设计：
1. **第一段：低延迟海量漏斗——多路召回策略体系（Retrieval）**：处于该管线的最粗段开口。任务是以几十万/几百甚至更高量级的文本碎片基数中将最有潜质嫌疑的 Top-100/200 过滤出境。要快要不含糊又要包容多类偏科，同时开搞 BM25全文相关倒排 和 ElasticSearch 或 MILVUS / FAISS 等近似高效紧邻（ANN算法库 HNSW 或分发），讲究海平面捞捕。
2. **第二段：极精密算力瓶颈——重排序融合与净化打分布道（Rerank / Filter）**：上游捞出来的数据是各异并带有大偏杂乱水份甚至打分标准体系根本不在同一个标度的“脏”大团。在这个隘口需要专门部署专门针对输入双向 Attention 的 Cross-Encoder 模型强行上算力将所有的侯选条目成对逐字精雕考量，将 Top 100 杀戮成 Top 5 / 10 极致极核浓缩资料。也在这里掺杂各类拦截硬逻辑、租户打分筛选过滤网、时序打折降级卡点手段。
3. **第三段：格式化成文与脑力聚合——提示词发散体系生成（Generation）**：拿着被前面环节洗剥干净绝对放心喂饱的高权重上下文提示大串内容（Context），用严密的框架级防幻觉带引用声明体系约束好最终生成提示词交送给推理脑力 LLM 发话打字。

**Q20: 介绍一下function calling和MCP**
- 难度：⭐⭐
- 岗位：开发岗重点
- 标签：#Function Calling #MCP
- 公司：字节、阿里

**【参考答案】**
这是从 RAG 通向行动派（Action-Oriented）Agent 的抓手工具：
- **Function Calling（函数调用机制）**：由大模型原厂商（如 OpenAI 牵头）主导推行。它是在用户发请求时带上一份系统里能调用的函数描述名册目录（用 JSON schema 声明参数）。模型不写代码不执行系统操作，而是模型通过思考得出“要不你这儿去走调用一个这个函子查查”再将包含该意图的标准化携带入参返回给中间件，而后由你的代码拿到结果调了真的物理机函数拿回值再二抛给模型。它是原生的模型能力边界和基建接口。
- **MCP（Model Context Protocol 模型上下文开源接入协议）**：这算是针对开发大盘痛点做的通用集成标准与基建组件，由 Anthropic 力邀推动。以往大家各做各大模型外设函数极其繁琐，写成乱如麻的针对不同数据端的绑定代码桥和脏配置；但 MCP 直接想要充当大模型界限里的通用水路总渠：一端连接各大 LLM Agent，一端规范接入 Github本地仓库/数据库/各类企业级后端资源。你使用 MCP-server 端协议声明规范挂起本地环境要素后所有标准 IDE 或 AI 都可如吃自助餐一般顺滑透明直接调度抓走。

**Q21: 在高并发查询 Agent 系统中，你会如何优化召回和生成阶段的延迟？**
- 难度：⭐⭐⭐
- 岗位：开发岗重点
- 标签：#高并发 #延迟优化
- 公司：字节(真题)

**【参考答案】**
在高并发请求下保持吞吐与低延时的硬核系统工程设计实践往往涵盖以下维度手段：
1. **召回阶段（高宽带IO和算法压榨并济）**：
   - 启用高效语言与异步协程（Go / Asyncio）把串行意图改写重写分路改成无依赖分支的全多路并发。
   - 大量部署在向量搜索器前排使用高度命中的缓存拦截层例如依靠开源大盘架构里面的极其巧妙设计的 Semantic Cache 针对近似问句短时间接兜。

   - **向客户端侧提供基于流式的返回架构（SSE / Websocket Streaming）**，这是最好的延迟伪装，TTFT（Time To First Token）压住在几百毫秒体感即优秀满格消除焦躁。
   - 彻底下潜大基座端侧：大厂一定会选择极速推断基建集群配置如 vLLM/TensorRT-LLM 接管替代原生端进行组批发车推断；利用好 PagedAttention 的内核优势实现显存零碎片利用率极大化压低 KV 空间以及建立类似 Prompt Caching （提示词状态大缓存共享池避免系统大开头词复读耗费大量运算算力阶段）。

**Q22: 如果让 agent 调用搜索引擎，如何避免无关结果影响回答？**
- 难度：⭐⭐
- 岗位：通用
- 标签：#结果过滤 #相关性
- 公司：字节(真题)

**【参考答案】**
直接拿引擎裸接口搜内容常常面临极度掺杂发散，影响模型回复聚焦。解决途径如下方案手段进行多维兜网限制控制：
1. **前馈清洗改口（Query Transformation & Sanitization）**：你发一句闲话进 Google 不会出来好东西，必须在发搜索动作前借助大基座做或者规则进行关键字 SEO 定位词重塑动作，洗出最干净的高效引擎搜索字段结构式表达式。
2. **域名特防拦截及定向兜底设置（Allowlist Domain Routing）**：强制携带 `site: ` 等强约束前置指定抓捕区域配置只允许官方机构论坛或高优域名字段被放入引擎抓取队列内摒弃农场污染域。
3. **后置抓获清洗重排网卡口（Lightweight Local Reranker）**：抓回带有数十片段 Snippet 或者全文内容的时候立刻通过内置小极速打分器（比如轻量 BGE-Reranker 后处理过阀过滤分数低下杂散不达标页的文本，不污染生成端。
4. **防御自控提示词加固防御屏障（Strict System Prompt Defensive Enclosures）**：进入生成阶段强塞极其冷峻严格系统提示：“以下是系统强制搜索引擎随机附送的资料摘要存在巨大概率部分段与主干无关的扰乱噪声垃圾片段内容，请极力认真挑选，只可参考针对本问有实质帮衬的相关信息并无杂废话只使用其作答不要受到旁杂诱发幻想。”
